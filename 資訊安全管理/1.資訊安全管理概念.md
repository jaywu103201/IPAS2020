 ##  1.1.1.CIA  
 # 機密性 (Confidentiality) 
 ```
 資訊不得被[未經授權]之個人、[實體]或[程序]所取得或揭露的性質。
 避免未經授權的使用者[有意]或[無意]的揭露資料內涵
 例如資料於網路傳送時被[攔截][竊取]，或公司[不小心]公佈不該公佈的訊息均是違反資料的機密性
 [機密性]資料傳遞和儲存時務必[加密]處理
 ```
 
 # 完整性 (Integrity)
 ```
 避免[非經授權]的使用者或處理程序[篡改]資料
 所使用的文件經[傳送]或[儲存]過程中必需證明其內容並未遭到[竄改]或[偽造]才能稱為[完整性]
 [md5@sun] [sha1@sum]
 ```
 
 # 可用性 (Availability )
 ```
 讓資料隨時[保持]可用狀況
 已授權實體在[需]要時[可存取]與使用資訊之性質
 系統的高度可用性通常指的是[必需]確保[不能中斷]服務
 ```
 # 鑑別度 (Authenticity)
```
確保使用者登入時有適當的驗證程序
```
# 可靠度 (Reliability)
```
確保作業執行皆有一致結果
```
# 可歸責性  (Accountability)
```
確保使用者執行任何動作均有適當的軌跡可追蹤至執行者。
```
## 1.1.2.各種破壞CIA的情境
```
破壞機密性  (Confidentiality)
 -被駭客入侵查看公司機密(未經授權)

破壞 (Integrity)
 -學生駭入學生網站竄改成績(更改檔案內容)
 
破壞 可用性 (Availability )
 -被DDos攻擊 [無法]連線  (無法使用)
 分散式阻斷服務攻擊（distributed denial-of-service attack)
```

## 重要資安概念 

# 邊界與分類(Boundary and classification) 
```
 1.防火牆、VPN、ID&P侵偵測與防禦（Intrusion Detection and Provention) 閘道防毒
```

# 職務區隔(Segregation of duties, SOD) 
```
給予職務執行上所必須知悉之內容，並設定明確的權責歸屬
個重要的工作或職責分散給多個人做保管，以防止這個重要資訊被洩露出去，即使被攻擊只有取得部份的資訊
特定業務責任的個人，與負責審查和核准該責任之相關動作者，不可以是同一人
球員不可兼任裁判
```

# 縱深防禦(Layered defense, defense in depth) 
```
以多層電腦安全技術去減輕被駭客入侵的風險
```
# 單一脆弱點(Single point of failure, SPOF) 
```
是指系統中一旦失效，就會讓整個系統無法運作的部件，換句話說，單點故障即會整體故障
```
# 阿奇里斯腱(Achilles heel) 
```
1.致命的弱點
```
# 木桶理論(Bucker principle） 
```
1.木桶的最大裝水量，取決於最短的一根木材，若其中有一根木板比其他的短，那麼木桶裝滿水的高度就是這根短木板的長度
```
# 僅知原則(Need to know)
```
1.完成一件任務所需要知道的內容及所需要知道的流程，僅給予完成工作的最少量的應知事項即可。
```
## 1.2.1.資訊安全管理系統（Information Security Management System, ISMS）
# 1_2_資訊安全管理系統(ISMS)
```
是一套有系統地分析和管理資訊安全風險的方法。

要達到100% 的資訊安全是一種過高的期望，資訊安全管理的目標是透過控制方法，把資訊風險降低到可接受的程度內。
```

# 資訊安全管理系統
```
ISO27001 資訊安全管理是[國際標準]。它概述了如何[建置]及[獨立]稽核驗證的資訊安全管理系統，因而降低被非法或未經允許使用的可能性。

```

# 1.2.2.導入ISMS的目的
```
ISO27001:2013內容包含了本文，及附錄的35項控制目標 (共114項控制措施)，
是一相當具有依循價值的標準，若能導入ISO27001此一國際標準，將有助從法遵的合規性、技術及管理的有效性來對資訊安全作出整體的思考。


企業在面對聯網設備數量增加或委外業務增加，資安風險大幅提升的情況下，藉由導入 ISMS強化資安風險盤點、軟體安全發展生命週期 (Secure Software Development Life Cycle，SSDLC)、
委外管理及資安事故處理等各控管面向的規劃與執行，有助降低面臨例如產線中毒、資料外洩及營運中斷等資安事件發生的可能性及衝擊，並提升客戶及關注方對企業穩健營運的信心。
```
# 1.2.3.導入ISMS的過程與程序
```
規劃(plan)(建立)
   1.建立全景(考慮事項、基本準則、範疇、組織)
   2.風險評鑑
   3.發展風險處理計畫
   4決定風險接

執行(Do)(實作與操作) 
   5.實施風險處

檢查(cheak)(監控與審查ISMS) 
   6.持續監控與審查風
行動(action)(維護與改進 )
   7.維護與改進資安風險管理程
```

# 1.2.4.導入ISMS的關鍵注意事項
```
導入任何有效的管理體系，其實與組織再造(Process Reengineering)的精神並無差異，都要對組織任務編組/技能/專長與資源投入進行分析與支配。因此用電腦技術的眼光來看待資訊安全管理，僅能描述事實的某一部分
```
